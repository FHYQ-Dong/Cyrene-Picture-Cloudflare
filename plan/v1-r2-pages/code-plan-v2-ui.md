## 3. 信息架构与页面规划

## 3.1 路由与职责

1. `public/index.html`（展示页，默认首页）

    - 顶部 Blog 标题栏。
    - 右上角“上传图片”按钮（跳转 `upload.html`）。
    - 图片瀑布流（卡片）+ 页面式加载（下一页）。
    - 分组维度：按日期分组，分组内可按上传者聚合。

2. `public/upload.html`（上传页）

    - 上传者昵称输入框（可空）。
    - 文件选择、Turnstile、人机验证状态。
    - 上传进度与结果回显。

3. `public/image.html`（详情页，保留）
    - 单图预览 + 元数据（含上传者昵称）。

## 3.2 导航行为

-   展示页右上角固定按钮：`上传图片`。
-   上传页顶部提供返回按钮：`返回展示`。

## 4. 可配置项规划（标题/主题）

推荐采用“前端配置对象 + CSS 变量”方案：

1. 文案配置
    - `siteTitle`：默认 `昔涟`
2. 颜色配置
    - `titleColor`：默认粉色（例如 `#e86aa6`）
    - `titleStripeColor`：默认淡粉半透明（例如 `rgba(255, 182, 212, 0.42)`）
3. 扩展项（可选）
    - `subtitle`
    - `themeBg`

配置优先级建议：

-   运行时配置（未来可接 `/api/site-config`） > 内置默认值。

## 5. 数据模型与迁移规划（新增上传者）

## 5.1 D1 表结构变更

对 `images` 表新增字段：

-   `uploader_nickname TEXT NOT NULL DEFAULT '093'`

新增索引建议：

-   `idx_images_uploader_created_at (uploader_nickname, created_at DESC)`

## 5.2 兼容存量数据

-   历史记录没有上传者昵称时，统一回填为默认昵称 `093`。
-   API 层兜底：若数据库返回空值，响应前兜底成默认昵称。

## 6. API 契约调整规划

## 6.1 `POST /api/upload-url`

-   保持现有上传逻辑。
-   新增可选请求字段：`uploaderNickname`（仅做格式校验与回传上下文，不影响签名逻辑）。
-   校验规则：
    -   去首尾空白。
    -   长度 `1-24`（可配置），超出截断或报错（二选一，建议报错更可控）。
    -   空值时不报错，后续走默认昵称。

## 6.2 `POST /api/upload-complete`

-   新增请求字段：`uploaderNickname`（可选）。
-   入库时写入 `uploader_nickname`：
    -   若空：写默认昵称。
    -   若有值：写标准化后的昵称。

## 6.3 `GET /api/list`

新增查询参数：

-   `groupBy`: `date | uploader | none`（默认 `date`）
-   `uploader`: 按昵称筛选（可选）
-   保留 `cursor/limit` 分页

响应扩展：

-   `items[]` 增加 `uploader_nickname`
-   增加 `groups[]`（可选，后端聚合模式）

## 6.4 `GET /api/image/:id`

-   返回中增加 `uploader_nickname` 字段。

## 10. 风险与缓解

1. 昵称污染（脚本/超长/非法字符）
    - 缓解：后端严格长度与字符白名单校验。
2. 瀑布流性能波动（大量图片）
    - 缓解：分页上限 + 图片懒加载 + 卡片骨架屏。
3. 分组分页一致性
    - 缓解：统一使用 `created_at + image_id` 作为稳定游标。
4. 旧数据无昵称
    - 缓解：迁移回填 + API 兜底默认昵称。
