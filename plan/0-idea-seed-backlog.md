### [Idea-008] 基于图片哈希的秒传与元数据关联

-   状态：已转入 v5 代码实现计划（见 `plan/v1-r2-pages/code-plan-v5-hash-instant-upload.md`）
-   背景/动机：重复图片上传会重复占用带宽与存储，上传体验与成本都不理想。
-   目标用户：重复投稿用户、转载整理用户、高频上传用户。
-   核心价值：命中已存在图片时可跳过对象上传，提升速度并降低存储冗余。
-   MVP 范围（最小可行）：上传前/上传后计算图片内容哈希（如 SHA-256），若命中已存在对象则直接复用对象并仅写入新的元数据记录。
-   暂不做：感知哈希（pHash）近似去重、跨格式归一化去重、客户端分片哈希。
-   可能风险：

    -   哈希计算增加 CPU 耗时；
    -   误把“同图不同上传语义”简单覆盖为同一记录；
    -   并发上传同图时可能出现竞态写入。

-   依赖条件：

    -   数据库支持按哈希索引查询；
    -   上传流程支持“复用旧对象 + 新建元数据关系”；
    -   需要定义“对象唯一性”与“上传事件唯一性”分离模型。

-   预计复杂度：中-高
-   下一步：

    -   先设计两层模型：`image_objects`（唯一内容对象）+ `image_upload_events`（每次上传事件）；
    -   明确当哈希命中时如何记录新的上传者、上传时间、来源批次并保持可追溯。

### [Idea-009] 展示界面自动根据图片尺寸做 UI 适配

-   状态：已转入 v6 代码实现计划（见 `plan/v1-r2-pages/code-plan-v6-adaptive-gallery-progress-ui.md`）
-   背景/动机：当前列表与详情在横图、竖图、超长图场景下展示一致性不足，影响浏览体验。
-   目标用户：移动端访客、连续浏览用户、对版面观感敏感的用户。
-   核心价值：根据图片宽高比自动选择更合适的展示样式，减少裁切失真与留白浪费。
-   MVP 范围（最小可行）：
    -   上传或回填阶段记录图片宽高；
    -   前端按比例分组（横图/竖图/方图）应用不同卡片样式；
    -   详情页按图片方向自动调整容器与操作区布局。
-   暂不做：AI 智能构图裁剪、自定义布局编辑器、用户手动覆盖布局。
-   可能风险：
    -   不同屏幕尺寸下样式规则复杂度提升；
    -   极端比例图片（超长图）仍可能出现滚动体验问题；
    -   现有 CSS 与瀑布流逻辑改动范围较大。
-   依赖条件：
    -   元数据中可稳定获取 `width/height/aspect_ratio`；
    -   前端渲染层支持按比例分类与样式切换；
    -   需要补充跨端视觉回归清单。
-   预计复杂度：中
-   下一步：
    -   先定义比例区间阈值（如 `<=0.8` 竖图，`0.8~1.2` 方图，`>=1.2` 横图）；
    -   输出展示页与详情页的适配规则表（桌面/移动端各一版）。

### [Idea-010] 上传过程由文字调试改为进度条可视化

-   状态：已转入 v6 代码实现计划（见 `plan/v1-r2-pages/code-plan-v6-adaptive-gallery-progress-ui.md`）
-   背景/动机：当前上传过程主要通过文本/JSON 调试信息反馈，普通用户阅读成本高，批量上传时可视性更差。
-   目标用户：批量上传用户、移动端用户、非技术用户。
-   核心价值：以进度条替代调试文案，直观展示上传状态并降低认知负担。
-   MVP 范围（最小可行）：
    -   单图上传时显示单一进度条（预检/上传/完成阶段）。
    -   批量上传时按文件显示多条进度条，每条独立状态与百分比。
    -   所有进度条外层加入固定高度滚动窗（纵向可滚动），避免挤占页面主视区。
    -   人机验证转圈区域下方不再显示 debug 红色框与“人机认证成功”提示。
-   暂不做：实时网速估算、时间剩余预测、拖拽排序与分组折叠。
-   可能风险：
    -   浏览器原生 `fetch` 对上传进度支持有限，需补充兼容方案（如 `XMLHttpRequest`）。
    -   进度状态同步复杂，可能出现 UI 状态与后端结果短暂不一致。
    -   批量高并发时频繁刷新进度条可能影响前端性能。
-   依赖条件：
    -   前端上传模块支持按任务维度维护状态机与百分比。
    -   样式层需提供进度条组件与滚动容器组件。
    -   批量接口返回逐项结果，便于最终状态收敛。
-   预计复杂度：中
-   下一步：
    -   明确上传阶段映射规则（`hash-checking`/`preparing`/`uploading`/`finalizing`）；
    -   输出上传页组件草图：进度条行、状态徽标、滚动窗尺寸与响应式策略。

### [Idea-011] 管理员接口新增“删除图片”能力

-   状态：已转入 v8 代码实现计划（见 `plan/v1-r2-pages/code-plan-v8-admin-delete-image.md`）
-   背景/动机：当前管理员仅能做缩略图补偿等维护动作，遇到违规内容、误传测试图或版权投诉时，缺少统一删除入口。
-   目标用户：站点管理员、内容审核与运维人员。
-   核心价值：通过受控管理接口快速下架并清理图片，减少人工 SQL / R2 手工操作风险。
-   MVP 范围（最小可行）：
    -   新增管理员删除接口（如 `POST /api/admin/delete-image`）。
    -   支持按 `image_id` 删除单张，返回删除结果与审计信息。
    -   新增批量删除接口（如 `POST /api/admin/delete-images`），支持逐项结果返回。
    -   软删除元数据（`images.status='deleted'`）并记录 `updated_at`。
    -   仅在对象无其他引用时删除对应 R2 原图与缩略图对象。
    -   复用现有管理员鉴权、限速、审计日志链路。
-   暂不做：回收站恢复、异步任务队列与多级审批流。
-   可能风险：
    -   误删导致内容不可恢复；
    -   引用计数不一致时可能误删共享对象；
    -   接口滥用带来高风险破坏。
-   依赖条件：
    -   现有 `admin` 鉴权机制（`ADMIN_API_TOKEN`）与审计表可复用；
    -   `image_objects.ref_count` 与上传事件关系准确；
    -   明确“软删元数据”与“物理删对象”的执行顺序和失败补偿策略。
-   预计复杂度：中-高
-   下一步：
    -   输出删除流程时序图（鉴权 → 查引用 → 软删元数据 → 条件删对象 → 审计）；
    -   先实现单图删除 dry-run 模式（只返回将要删除的资源，不落库）；
    -   定义错误码（无权限、对象仍被引用、R2 删除失败、审计写入失败）。

### [Idea-012] 批量上传超过 20 张时后续任务卡住（待修复）

-   状态：已转入 v10 代码实现计划（见 `plan/v1-r2-pages/code-plan-v10-batch-upload-over20-stuck-fix.md`）
-   后续优化：已转入 v11 计划（见 `plan/v1-r2-pages/code-plan-v11-batch-turnstile-session-optimization.md`）
-   背景/动机：当前批量上传在大于 20 张时会出现“前 20 张完成、后续卡住”的错误成功反馈，影响可用性与数据一致性认知。
-   目标用户：高频批量上传用户。
-   核心价值：确保批量上传在大批次下“全部可处理、状态准确、结果可信”。
-   复现条件：
    -   一次选择 `>20` 张图片（例如 30 张）；
    -   点击批量上传后，前 20 张成功；
    -   第 21 张及后续进度停留在“申请上传”；
    -   页面最终显示“上传成功”但实际未完成全量。
-   初步根因（待确认）：
    -   后端批量 complete/prepare 存在 `MAX_BATCH_ITEMS = 20` 截断；
    -   前端结果汇总仅按已返回项计数，未将“未处理尾项”标记为失败或待处理。
-   MVP 范围（最小可行）：
    -   去除或显式处理 20 条上限（分片批次处理）；
    -   保证所有选中文件都有终态（`success/failed/canceled`）；
    -   若触发上限，UI 明确提示并自动续传下一批，而非静默卡住。
-   暂不做：动态并发自适应、断点续传。
-   可能风险：
    -   直接放大单次批量上限可能触发限流；
    -   分片后错误聚合逻辑复杂度上升。
-   依赖条件：
    -   批量 prepare/complete 接口契约调整；
    -   前端队列状态机支持“分批串联 + 总批次统计”。
-   预计复杂度：中
-   下一步（明日）：
    -   明确上限策略：单批固定上限 + 自动分片（推荐）；
    -   补自动化回归用例：`files.length = 21/50`；
    -   修复后验证“无卡住 + 汇总计数准确 + 错误可解释”。

### [Idea-013] 支持音频展示和上传

-   状态：想法池（待评审）
-   背景/动机：当前站点仅支持图片内容，无法承载 BGM、语音片段、音频投稿等场景。
-   目标用户：需要上传/分享音频内容的创作者与访客。
-   核心价值：扩展站点内容形态，支持“图 + 音频”并行展示，提升投稿覆盖面。
-   MVP 范围（最小可行）：
    -   上传侧支持常见音频 MIME（如 `audio/mpeg`、`audio/wav`、`audio/ogg`、`audio/mp4`）。
    -   列表与详情支持音频卡片与播放器（播放/暂停/时长显示）。
    -   元数据新增 `media_type`、`duration`（可回填）。
    -   与现有限流、配额、审计、管理员删除能力兼容。
-   暂不做：歌词同步、波形编辑、转码队列、音频指纹去重。
-   可能风险：
    -   音频文件体积较大，配额与带宽压力上升；
    -   浏览器兼容与编解码格式差异增加失败率；
    -   内容审核复杂度高于图片。
-   依赖条件：
    -   上传接口 MIME 白名单与前端选择器扩展；
    -   数据库 schema 增加音频元数据字段；
    -   前端组件新增播放器与媒体类型渲染分支。
-   预计复杂度：中-高
-   下一步：
    -   先输出 v11 可行性与 schema 迁移草案；
    -   定义媒体统一模型（`image|audio`）与兼容策略；
    -   规划最小播放器 UI 与移动端交互约束。
